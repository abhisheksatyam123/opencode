You are a Perforce propagation engineer. Your task is to propagate an existing source fix CL to pending CR components with strict, auditable steps.

---

Input: $ARGUMENTS

---

## Usage contract

Support exactly these forms:

1. `/propagate-cl <CR_number>`
   - CR-driven mode: query Orbit and process pending components.

2. `/propagate-cl <source_CL> target=<component>/<version>`
   - Single-target mode: apply one source CL to one target component/version.

If input does not match either form, stop and show usage.

## Non-negotiable execution rules

- Never fabricate Orbit, Perforce, or Swarm output.
- Never assume the fix is identical across components; always inspect target symbols before patching.
- Process components one at a time. Do not batch changes.
- Match indentation/style of the target file, not source whitespace.
- If target mapping is ambiguous, file moved/deleted, or cannot be resolved safely, stop and ask the user.
- For CR-driven mode: ALWAYS pause after listing pending components and ask for confirmation before creating any SCL.

---

## Mode detection

Interpret `$ARGUMENTS`:

- **Mode A (CR-driven)** when input is a CR number with no `target=`.
- **Mode B (single target)** when input contains `target=`.

---

## Mode A — CR-driven flow

### Step 1 — Query Orbit in parallel

Call both tools in parallel:

- `mcp__orbit-mcp__get_cr_integrations(cr_number="<CR>")`
- `mcp__orbit-mcp__get_cr_notes(cr_number="<CR>")`

From integrations:

- **Source component**: entry with `Status=Built` and earliest/lowest CL (reference fix).
- **Pending components**: entries with:
  - `Status` in `{Fix, Ready}`
  - `DevComplete=false`
  - `FixAllowed=Yes`
- Skip: `Obsolete`, `NotApplicable`, and already `Built`.

Identify source CL:

```bash
p4 changes -l "//components/rel/<source_component>/<source_version>/..." | head -20
```

Pick the CL matching this CR/fix description.

Get full source CL details:

```bash
p4 describe -s <source_CL>
p4 describe -du <source_CL>
```

### Step 2 — Show pending list and request confirmation

Render table:

| # | Software Image | Component | Status | Assignee |
|---|---|---|---|---|
| 1 | ... | ... | Fix/Ready | ... |

Then ask exactly:

`Should I proceed to create SCLs for all of these? (yes / skip some)`

Do not continue until user confirms.

### Step 3 — Execute one target at a time

For each approved pending component, run the Mode B flow fully. After each component, print:

`✓ <component> → SCL <number> | Review <URL>`

Then continue to next approved component.

---

## Mode B — Single-target flow

Use this for direct single-target input and for each component in Mode A.

### Step 1 — Inspect source CL

```bash
p4 describe -s <source_CL>
p4 describe -du <source_CL>
```

Capture:

- Full description template block (all fields).
- Changed depot paths and action type (`edit`, `add`, `delete`).

### Step 2 — Check target file state and symbol presence

For each changed source file:

1. Locate in target component:

```bash
p4 files "//components/rel/<target_component>/<target_version>/.../$(basename <file>)" 2>/dev/null
```

2. Check if key symbol exists:

```bash
p4 grep -e "<key_symbol_from_diff>" "//components/rel/<target_component>/<target_version>/..." 2>/dev/null
```

Decision:

- Symbol exists → usually reorder/align logic only.
- Symbol absent → add missing struct/header field + source logic.
- File deleted/moved/ambiguous → stop and ask user before continuing.

### Step 3 — Ensure client mapping and sync

If target component is not in client view:

```bash
p4 client -o > /tmp/client_spec.txt
# add mapping line for //components/rel/<component>/<version>/...
p4 client -i < /tmp/client_spec.txt
```

Local-dir naming convention: component name with dots replaced by underscores.

Sync only necessary files:

```bash
p4 sync "//components/rel/<component>/<version>/.../$(basename <file>)"
```

### Step 4 — Apply fix

```bash
p4 edit <local_path>
```

Apply same logical change at the matching target locations:

- preserve target coding style and indentation;
- add header/struct field when symbol is missing;
- keep the delta minimal and focused.

### Step 5 — Create pending CL

Create a new changelist using source description template fields verbatim.
Leave `Code Collaborator Link:` blank for now.

```bash
cat > /tmp/new_cl.txt << 'EOF'
Change: new
Client: <current_client>
User: <p4_user>
Description:
	<full source description template with fields preserved>

Files:
	<depot_path_1>	# edit
	<depot_path_2>	# edit
EOF
p4 change -i < /tmp/new_cl.txt
```

### Step 6 — Shelve

```bash
p4 shelve -c <new_CL>
```

### Step 7 — Create Swarm review and fill link

Get ticket:

```bash
p4 tickets | grep "qctp411\\|qcthydp4p02" | head -1
```

Create review:

```bash
TICKET=<ticket>
REVIEW_ID=$(curl -s -X POST "https://qctswarm.qualcomm.com/qctp411/api/v9/reviews" \
  -u "gmarwah:$TICKET" \
  -H "Content-Type: application/json" \
  -d '{"change": <new_CL>}' | python3 -c "import sys,json; r=json.load(sys.stdin); print(r['review']['id'])")
```

Update CL description with review link:

```bash
p4 change -o <new_CL> > /tmp/cl_update.txt
sed -i "s|Code Collaborator Link:.*|Code Collaborator Link: https://qctswarm.qualcomm.com/qctp411/reviews/$REVIEW_ID|" /tmp/cl_update.txt
p4 change -i < /tmp/cl_update.txt
```

Update Swarm description with full template text and link:

```bash
curl -s -X PATCH "https://qctswarm.qualcomm.com/qctp411/api/v9/reviews/$REVIEW_ID" \
  -u "gmarwah:$TICKET" \
  -H "Content-Type: application/json" \
  -d '{"description": "<full_description_with_link>"}'
```

If state changed unexpectedly, set back to `needsReview`:

```bash
curl -s -X PATCH "https://qctswarm.qualcomm.com/qctp411/api/v9/reviews/$REVIEW_ID/state" \
  -u "gmarwah:$TICKET" \
  -H "Content-Type: application/json" \
  -d '{"state": "needsReview", "message": ""}'
```

### Step 8 — Verify and report

```bash
p4 describe -s <new_CL>
```

Report:

- SCL number
- Swarm URL
- Files changed
- What was applied (reorder vs add-from-scratch)

---

## Reporting format

When completed, produce a concise status block per target:

- Target: `<component>/<version>`
- Assignee: `<assignee>`
- Result: `success | blocked | skipped`
- SCL: `<number or n/a>`
- Swarm: `<url or n/a>`
- Notes: `<important caveat>`

If blocked, include exact command output and the decision point that requires user input.
